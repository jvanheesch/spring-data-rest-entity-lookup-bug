Incorrect serialization - changing a property from @Transient to @Embedded breaks serialization (deserialization still works as expected)

This is a rather complicated issue, so this explanation will clarify the reproducer (see reference url).

Context:
Jackson's Jdk8Module contains support for (de)serializing objects of type Optional<T>.
By default, <code>@JsonInclude(JsonInclude.Include.NON_NULL) Optional<String> someOptional</code> in java will correspond with one of three things in json:
- someOptional: "x", when someOptional is an Optional containing the String "x"
- someOptional: null, when someOptional is the empty Optional
- absent in json (= undefined), when someOptional is null.
This behavior is demonstrated by the following two tests in the reproducer:
givenAStringOptionalOwner_whenSerializing_thenEmptyOptionalLeadsToNullAndNullLeadsToAbsentProperty
givenAStringOptionalOwner_whenDeserializing_thenNullLeadsToEmptyOptionalAndAbsentPropertyLeadsToNull

Goal:
I wanted this exact behavior (i.e. distinction between null and empty Optional) for my own container/wrapper class (*).
I defined a simple class StringContainer, which is basically a wrapper around a String.
I created a jackson Module (StringContainerModule) based on jackson's Jdk8Module, as StringContainer is similar to Optional<String>.
I registered this module with Spring Data Rest (see SpringRestConfigurer)
I created an @Entity StringContainerOwner, which has 3 fields of type StringContainerOwner.
The fields are annotated with @Transient, such that they are not persisted.
The serialization/deserialization behavior works as expected, as demonstrated by the following two tests:
givenAStringContainerOwner_whenSerializing_thenEmptyContainerLeadsToNullAndNullLeadsToAbsentProperty
givenAStringContainerOwner_whenDeserializing_thenNullLeadsToEmptyContainerAndAbsentPropertyLeadsToNull

I now wanted to persist these StringContainer fields to my db (in-memory h2).
(Side note: I actually want the StringContainer to be @Entity rather than @Embeddable. I noticed the same serialization error in both cases,
and made a reproducer for the @Embeddable case, as it's closer to the working @Transient example - no need to add an @Id etc.)
I created a new class StringContainerEmbeddedOwner.
This class contains a single field of type StringContainerOwner.
This field is now annotated with @Embedded, whereas earlier all fields were annotated with @Transient.
This causes the serialization to fail, as demonstrated by






(*)
The reason I wanted this behavior is as follows:
In our crud application, we have our DomainObject which has many nullable fields.
Business rules dictate that many fields are only applicable/meaningful under certain circumstances.
Under any other circumstances, users should not be able submit values for these inapplicable fields.
"Not applicable" and "blank" are different concepts.
Often, both are modelled as "NULL" in the domain, but I was wondering if I could model them separately.
Javascript supports this natively with undefined vs null, so I wanted to see if I could design my resources this way.
To accomplish this, StringContainer should become an Entity rather than an Embeddable.
However, I noticed that I get the same error with both Embeddable and Entity.
